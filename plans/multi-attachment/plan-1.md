# Plan: Multi-Attachment Selection in Chat

## Objective

Allow users to select multiple Google Drive files as attachments in the chat input.
- Pill preview shows `<filename> + x others`
- Hovering on `x others` reveals a tooltip listing all selected filenames
- All selected files are passed as separate `drive_file` content blocks in the message
- Selection persists across page reloads via conversation's active files

## Current State

| Layer | What exists |
|---|---|
| Frontend state | `inputAttachment: TextAttachment \| null` (single) |
| Frontend hook | `useChatInput.ts` adds one drive file at a time |
| Frontend stream | `useChatStream.ts` emits one `drive_file` content block |
| DB model | `Conversation.active_drive_file_id: int \| null` (single FK) |
| API | `PUT /active-file`, `DELETE /active-file` (singular) |

## Target State

| Layer | What changes |
|---|---|
| Frontend state | `inputAttachments: TextAttachment[]` (array) |
| Frontend pill | Shows first filename + `+ N others` with hover tooltip |
| Frontend hook | Drive picker appends/removes from array |
| Frontend stream | Emits one `drive_file` block per selected attachment |
| DB model | New `conversation_active_drive_files` junction table |
| API | New bulk `PUT /active-files`, `DELETE /active-files` endpoints |

---

## Scope Clarification

- Only Drive file attachments support multi-select. Local file uploads remain single-attachment (one per message, not persisted across messages). The pill component renders both types but only Drive files accumulate.
- The existing single-file endpoints (`/active-file`) must be kept intact and not removed — they are used by the existing `DELETE` path for clearing a single file. Migrate the frontend to the new bulk endpoints instead.
- `DriveFilePicker.tsx` must support selecting multiple files in one open session. The currently selected set should be visible so users can deselect individual items.

## Data Flow Summary (after implementation)

```
User opens DriveFilePicker
  → selects 3 files, clicks "Confirm"
  → frontend calls PUT /active-files { drive_file_ids: [1, 2, 3] }
  → backend validates ownership, replaces junction table rows
  → frontend sets inputAttachments = [TextAttachment(1), TextAttachment(2), TextAttachment(3)]

Pill renders: [icon] "report.pdf + 2 others [×]"
  hover "2 others" → tooltip: "slides.pptx\nnotes.docx"

User types "summarize these" and hits Send
  → useChatStream.buildUserMessages() creates:
      content: [
        { type: "drive_file", file_id: 1 },
        { type: "drive_file", file_id: 2 },
        { type: "drive_file", file_id: 3 },
        { type: "text", text: "summarize these" }
      ]
  → backend _resolve_drive_file_blocks() resolves each to RAG context
  → LLM receives all three contexts + user query
```

- Reordering selected files.
- Per-file remove in the pill (only remove-all is exposed in the pill; individual management is via re-opening the picker).
- Pagination or search inside `DriveFilePicker` — pre-existing behavior unchanged.
